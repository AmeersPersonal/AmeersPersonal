# TakeMeter: Raspberry Pi / Embedded Maker Discourse Classifier

AI201 Project 3 fine-tunes a text classifier that labels discourse quality and intent in Raspberry Pi and embedded maker communities.

## Project Goal

Online maker communities mix useful technical guidance, help requests, broad unsupported claims, and lightweight project reactions. TakeMeter classifies posts into those discourse roles so the community can better understand what kind of conversation is happening and where human review or follow-up might help.

## Label Taxonomy

| Label | Definition |
| --- | --- |
| `evidence_based_guidance` | Concrete, technically grounded advice using specific parts, commands, measurements, code, diagrams, or reproducible troubleshooting steps. |
| `implementation_question` | A specific build, bug, wiring, library, or deployment question with enough context for others to answer. |
| `unsupported_claim` | A broad recommendation, warning, ranking, or confident opinion without enough evidence or implementation detail to justify it. |
| `showcase_reaction` | A project update, celebration, reaction, or brief comment that does not ask for technical help or make a supported technical argument. |

Hard edge case: if a post is opinionated but includes concrete reproducible evidence, label it `evidence_based_guidance`. If the technical detail is vague or only decorative, label it `unsupported_claim`.

## Dataset

Required format:

```csv
text,label
"Use a logic-level MOSFET instead of driving a motor from GPIO.",evidence_based_guidance
```

The project requires at least 200 labeled examples. Target distribution is roughly 50 examples per label, with no label below 20% of the dataset if possible.

Collected dataset:

| Source | Rows | Notes |
| --- | ---: | --- |
| Raspberry Pi Stack Exchange questions | 50 | Public question posts labeled `implementation_question`. |
| Raspberry Pi Stack Exchange answers | 50 | Public answer posts labeled `evidence_based_guidance`. |
| Hacker News Algolia maker/embedded search results | 100 | Public comments/stories labeled into `unsupported_claim` and `showcase_reaction`, with additional technical/question rows available during collection. |

Label distribution:

| Label | Count |
| --- | ---: |
| `evidence_based_guidance` | 50 |
| `implementation_question` | 50 |
| `unsupported_claim` | 50 |
| `showcase_reaction` | 50 |

The generated CSV is `data/annotated_examples.csv`. It includes `text`, `label`, `source`, `source_url`, `author`, and `label_reason`. The notebook uses only `text` and `label`; the other columns are kept for attribution and auditability.

Labeling process:

The collection script uses source-aware labeling rules. Raspberry Pi Stack Exchange questions are labeled as implementation questions, and Stack Exchange answers are labeled as evidence-based guidance. Hacker News rows are labeled with stricter pattern rules for broad unsupported claims, short project reactions, technical guidance, and question-like posts. A sample review was performed after generation, and the unsupported-claim rule was tightened to avoid labeling simple appreciation comments as claims.

Difficult examples to discuss after training:

1. Broad recommendation comments that include a technical detail can sit between `unsupported_claim` and `evidence_based_guidance`.
2. Short comments such as "works great on my Raspberry Pi" can sit between `showcase_reaction` and `unsupported_claim` if they imply a recommendation.
3. Stack Exchange questions with self-contained reasoning may look like guidance, but they remain `implementation_question` when the main purpose is asking for help.

## Training Approach

The notebook fine-tunes `distilbert-base-uncased` using HuggingFace `Trainer`.

Default hyperparameters:

| Hyperparameter | Value | Reason |
| --- | ---: | --- |
| Epochs | 3 | A conservative default for about 200 examples; more epochs may overfit. |
| Learning rate | 2e-5 | Standard starting point for BERT-family fine-tuning. |
| Train batch size | 16 | Fits a Colab T4 GPU for this model size. |
| Eval batch size | 32 | Speeds evaluation without affecting training gradients. |
| Split | 70/15/15 | Keeps separate train, validation, and locked test sets. |

## Baseline

The zero-shot baseline uses Groq `llama-3.3-70b-versatile` on the same test set as the fine-tuned model. The prompt is defined in the notebook and requires the model to output exactly one label name.

## How To Run

1. Open `ai201_project3_takemeter_starter_clean.ipynb` in Google Colab.
2. Set runtime to T4 GPU.
3. Add `GROQ_API_KEY` in Colab Secrets and enable notebook access.
4. Run all cells.
5. Upload the annotated CSV when prompted.
